## What I Do

Most B2B SaaS documentation is organized around how a product was built. This creates
a gap between what users need to know and what the documentation tells them. That
gap generates confusion, support tickets, and potentially churn that never gets
attributed to the documentation that caused it.

I close that gap.

Before writing a single word, I read a product's documentation architecture the 
way a forensic analyst reads a crime scene. I identify the primary failure mode the
documentation is producing, map the entry states of different user cohorts, and find where the structure is working against the user rather than for them. Then I
build documents organized around what users need to accomplish, not around what the
product needs to explain.

The writing comes last. The architecture comes first.

---

## How I Work
I use AI as an execution partner and my analytical judgment as the architectural
layer that directs it. Every document in this portfolio started with a diagnostic reading of the product, a structural brief that defined the primary
failure mode to prevent and the user cohort to serve, and a pattern selection
chosen for a specific reason.

The framing note inside each portfolio folder explains those decisions. Reading
the framing note before the document will show you not just what I wrote but why
I built it the way I did.

## Background

My analytical foundation is in pricing forensics and organizational diagnosis. I
built Strategic Pricing Forensics (SPF), a methodology for diagnosing structural
pricing architecture failures in B2B SaaS companies using public signals without
internal data access.

That methodology transferred directly into documentation work. Reading a documentation system for organizational health signals, finding where the architecture fails the user, and connecting that failure to commercial consequences like churn and support volume, these are the same diagnostic moves applied to a different surface.

The result is technical writing that is not just clear. It is architecturally honest. It discloses constraints before users encounter them. It meets users at their cognitive state rather than at the product's feature list. And it is designed to make the documentation itself unneccessary over time, because users who understand a process deeply do not return to the documentation that taught it to them.
